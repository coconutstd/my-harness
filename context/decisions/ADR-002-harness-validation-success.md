## 상태

**Accepted**

## 배경

TASK-001을 통해 Worklog / Branching / Task Management 3개 하네스가 실제 개발 흐름에서 제대로 동작하는지 검증해야 했다.

---

## 결정

3개 하네스 모두를 실제 Next.js 기능 개발(About 페이지)로 테스트한 결과, **모두 정상 동작함을 확인했으며, 이 패턴을 지속 사용하기로 결정했다.**

### 검증 결과

| 하네스 | 기능 | 결과 |
|--------|------|------|
| Worklog | 커밋 후 맥락 기록 | ✅ 자동으로 `context/worklog/2026-04-09.md`에 커밋 섹션 기록됨 |
| Branching | 브랜치 생성/추적 | ✅ strategy.md 규칙 준수, `active-branches.md` 실시간 업데이트 동작 |
| Tasks | 상태 관리 및 동기화 | ✅ TASK-001 상태 변경 시 TASK 파일과 `_index.md` 자동 동기화됨 |

### 특히 유용한 점

1. **에이전트 행동 일관성**: 3개 하네스가 에이전트의 자동화된 행동 기준점 역할
2. **컨텍스트 선택적 로딩**: INDEX.md로 필요한 파일만 읽게 함
3. **작업 투명성**: 커밋-브랜치-태스크가 연계되어 작업 흐름이 명확함

---

## 대안

1. **Git 커밋 메시지만 사용**: 비용 절감이지만, 구조화된 맥락 기록 불가능
2. **외부 프로젝트 관리 도구(Linear, Jira)**: 강력하지만, 로컬 파일 기반 패턴과 충돌 가능
3. **Hook 기반 자동화**: 더 강력하지만, 초기 설정 복잡도 높음

→ 현재의 파일 기반 Harness가 간결성과 효율성의 최적점

---

## 결과 및 트레이드오프

### 장점
- ✅ 에이전트가 규칙을 일관되게 따름
- ✅ 파일 수정만으로 프로세스 변경 가능
- ✅ 버전 관리(git) 기반 이력 추적

### 비용
- 📝 매 작업마다 파일 수정 필요 (자동화 가능성은 있음)
- 📂 컨텍스트 파일 관리 필요
- 🔄 INDEX.md 동기화 필수

### 결론

파일 기반 Harness는 소규모 팀/개인 개발자에게 적합한 패턴이다.
향후 팀 규모 확대 시 외부 도구 도입 검토.
